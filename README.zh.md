# Hermes Agent 源码分析

> 18,610 次提交。约 64 万行 Python（还不算 72 万行测试）。一个主要由 AI 写出来的自我进化 Agent 框架。
>
> 我们把它整个读完、拆开——15 篇叙事式文档，中英双语。

这是对 [NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent) **v0.18.2**（tag [`v2026.7.7.2`](https://github.com/NousResearch/hermes-agent/releases/tag/v2026.7.7.2)，commit `9de9c25f6`，2026-07-07）的一份完整源码分析。它不是 API 手册，也不是使用教程，而是一次**架构拆解**——像拆开一台精密机器，看清每个齿轮为什么在那儿、和谁咬合、拿掉会坏在哪。

每篇文档都想同时做两件事：既帮你**深度理解架构**，又能**直接指导使用**。每章先讲使用指南（是什么、怎么用、怎么配、常见场景、排错），再进架构与实现，对每个子系统回答同一个问题：**它面对什么问题、做了哪些选择、为什么不用别的方案、出错时会怎样。**

## 分析对象

Hermes Agent 是 Nous Research 出品、MIT 许可的自我进化 AI Agent 框架。它同时是好几样东西：一个能用的 AI 助手、一个服务 **20 个插件平台 + 9 个内建平台**消息的统一网关、一个带 **69 个注册工具**的执行引擎、一个训练数据工厂，以及自 v0.18 起的跨平台桌面客户端（约 14 万行 TypeScript）。

有意思的是：其中相当大比例的代码，很可能是 AI 自己写的（[证据见第 00 章](docs/zh/00-项目全景.md)）。

## 这份分析是怎么做出来的

64 万行代码，一个人——或一个 AI——一遍读不准。每章都走一条多 Agent 流水线：主线（Opus）读源码、写草稿，专职审核 Agent（Sonnet）挑错。

```
主线 (Opus)            → 读源码，对照 v1（v0.11.0）深度基准写草稿
  ↓
深度审核 ‖ 文学性审核    → 并行：深度缺口 / 叙事质量
  ↓
事实审核               → 逐条对着实际代码验证
  ↓
主线 (Opus)            → 每个 ⚠️/❌ 二次验证 → 修正 → 定稿
```

一条硬规则贯穿始终：**审核发现的每一处，都必须再做一次独立代码阅读验证后才写进文档**——两次独立结论一致才作数。每个概念都过一遍 9 问模板（是什么 / 从哪来 / 在哪层 / 谁依赖它 / 怎么运作 / 解决什么 / 为何不用别的 / 出错怎样 / 能配什么），织进叙事，而不是堆成表格。

中文先写，再经一条专门的翻译审核流水线译成英文，配一份共享[术语词库](docs/TRANSLATION_GLOSSARY.md)——代码标识符、行号、Mermaid ID 逐字符原样保留，只译散文和注释。翻译这一遍还反向查出并修正了 2 处中文源里的错误。

方法论不是一开始就有的，而是在工作中从用户反馈里长出来的；完整演变过程记在工作日志里（中文，见下）。

## 几个值得知道的点

1. **相当大比例的代码很可能是 AI 写的**——而这个项目本身就是个 AI Agent。把 dogfooding 做到了极致（[第 00 章](docs/zh/00-项目全景.md)）。

2. **上帝文件的拆解真的奏效了。** `run_agent.py` 从 13,293 行（v0.11.0）缩到 **6,013 行**——`agent/` 子目录被逐步抽出来了。但巨物换了地方：`gateway/run.py`（20,719 行）和 `cli.py`（16,184 行）成了新的重量级（[第 02](docs/zh/02-Agent核心.md)、[05](docs/zh/05-网关层.md)、[10 章](docs/zh/10-交互界面与运行模式.md)）。

3. **安全模型对自己的边界很诚实。** `SECURITY.md` 区分了*真边界*（沙箱 / 进程隔离）和*启发式*（命令审批、脱敏、Skills Guard），并直言 prompt injection 和启发式被绕过**不在受理范围内**、也没有 bug bounty。把边界划清楚本身就是一种工程成熟度（[第 13 章](docs/zh/13-工程实践.md)）。

4. **Prompt Caching 的护栏比缓存本身更精巧。** 双注入（静态记忆进系统提示求命中、动态检索走用户消息保新鲜）+ 技能的三级渐进披露，是缓存友好架构的范本（[第 02 章](docs/zh/02-Agent核心.md)）。

5. **它确实能产出训练下一代自己的数据**——`batch_runner.py` 批量产轨迹 → `trajectory_compressor.py` 压进训练窗口。注意：RL *训练*环境已在 v0.14.0 移出主仓，所以第 12 章聚焦仍留在仓里的数据生成侧（[第 12 章](docs/zh/12-批量运行与轨迹生成.md)）。

## 相比 v0.11.0 变了什么

这份分析取代了更早一版对 v0.11.0 的分析（12 篇）。主要变化：

- **RL 训练基础设施移出**主仓（v0.14.0）——第 12 章从「批量运行与 RL」重定为只讲轨迹生成。
- **平台大迁移**——核心消息平台从 gateway 内建挪到插件（现为 20 插件 + 9 内建）。
- **多了个桌面客户端**（v0.17）——完整的 Electron + React 应用，配 Tauri 引导安装器，新增为第 14 章。
- **文档从 12 篇扩到 15 篇**，并且现在是**完整中英双语**。

## Agent 技能 —— 把这份专业度装进你的编码 agent

除了给人读,这份分析还打包成了一个 **Agent 技能**（[`.claude/skills/hermes-agent-expert/`](.claude/skills/hermes-agent-expert/)），让 Claude Code 这类编码 agent 能**直接用**它——在搭建 / 配置 / 调试 / 定制 / 二次开发 hermes-agent 时给出有源码依据的判断,并在需要时被指引到 hermes 真源码的正确位置。它的设计是一条 Tier 0→3 的渐进阶梯,终点是读真源码;其中约 200 个 `file:line`/符号/配置锚点都对钉定版本做过机制化校验。

# 行文标准：v1 深度分析

> 本文总结 v1 文档中验证过的行文方法和深度标准，作为 v2 写作的底线要求。

## 一、叙事结构：从问题出发，不从代码出发

v1 的每一篇都遵循同一个叙事逻辑：

```
问题 → 设计张力 → 选择 → 代价 → 替代方案
```

**反面**：先列出代码结构，再描述每个文件做什么（这是 v1 第一版被否掉的写法）。

**正面示例**（02-agent核心）：

> "每次调用模型 API，完整的消息序列都要从头发送。一个会话中系统提示可能占 5,000-10,000 token，但它在 20 轮对话中几乎不变——等于同样的内容付了 20 次钱。"

先让读者感受到问题的痛感，再引出 Prompt Caching 的机制。读者此时不需要被说服"这个模块值得了解"——他已经知道了。

## 二、每个概念的深度标准

v1 成熟阶段对每个核心概念的覆盖深度：

### 必须回答的 9 个问题

不需要逐个列为小节，但读者读完后必须能找到答案：

1. **是什么** — 一句话定义 + 一个类比（"如果 LLM 是大脑，工具是手脚，AIAgent 就是神经系统"）
2. **从哪来** — 什么情况下需要这个东西（"早期只需要一个 API key，但当支持 OAuth 之后……"）
3. **在哪里** — 精确到文件名:行号（`credential_pool.py:91`）
4. **依赖关系** — 谁创建它、谁使用它、谁拥有它（"池的创建由 CLI 层完成，Agent 只是消费者"）
5. **怎么工作** — 核心算法/流程，必须配图（状态机、流程图）
6. **解决什么** — 量化收益（"节省 75% 的 token 费用"）
7. **替代方案** — 为什么不用更简单的做法（"纯指数退避会产生惊群效应"）
8. **失败模式** — 出错了会怎样（"缓存完全失效不会崩溃，只是退回全量计费"）
9. **可配置性** — 用户能调什么（"config.yaml 的 credential_pool_strategies"）

### 深度参照

以 v1 的 02-agent核心 中"重试与退避"为例，一个概念的完整覆盖包括：

- 退避公式（`delay = min(base × 2^(attempt-1), max_delay) + jitter`）
- 为什么加 jitter（50 个 Gateway 会话同时重试的惊群问题）
- jitter 的随机 seed 设计（`time_ns ^ (counter × 0x9E3779B9)`，黄金比例位混合）
- 429 的分层处理（第一次不切凭证的原因——给 Provider 时间重置限流窗口）
- 与 Credential Pool 的协同关系
- 配套的 Mermaid 流程图

这不是过度细节——每一层都在回答"为什么"，而不是"是什么"。

## 三、图表不是装饰

v1 的图表使用规则：

1. **每个多步骤流程（≥3 步）必须有图**——文字描述流程容易让读者在第 4 步迷路
2. **每个状态转换必须有状态图**——以 Credential 的 ok→exhausted→refreshing 为例
3. **每个组件关系必须有架构图**——以 AIAgent 的四向协作（调用者、LLM、工具、持久化）为例
4. **图必须有标题**——"图：xxx"格式，告诉读者这张图在说什么
5. **图上的文字要有信息量**——不是简单的模块名，而是包含关键细节（"创建 ≤128 个 AIAgent（缓存）"）

## 四、代码引用是论据，不是正文

v1 的代码引用方式：

- **行号标注**：关键断言后面跟（`file.py:123`），让读者可以验证
- **代码片段**：只在解释算法或格式时贴代码，且长度不超过 10 行
- **不做代码导览**：不逐文件列出"这个文件有哪些函数"

**底线**：如果去掉所有代码引用和行号，文档的叙事仍然完整可读。代码引用是佐证，不是骨架。

## 五、具体案例标注

当用一个具体案例说明通用模式时，必须明确标注"以 X 为例"，避免读者误以为描述只适用于该特定场景。

**正面**："以 Anthropic 的 Prompt Caching 为例，它允许最多 4 个 cache_control breakpoint"
**反面**："Prompt Caching 允许最多 4 个 breakpoint"（读者会以为这是所有 Provider 的限制）

## 六、章节开头的定位块

每章开头必须有定位块，帮助读者快速判断"这章和我有没有关系"：

```markdown
> **本章定位**：`tools/` 目录（96 文件，68,250 行）+ `model_tools.py`（923 行）。
> **关键类**：`ToolRegistry`（`tools/registry.py`）、`handle_function_call()`（`model_tools.py`）。
```

## 七、v2 超越 v1 的方向

v1 的数字基于 v0.11.0，v2 初版基于 v0.14.0，2026-07 起已整体升级到 v0.18.2（tag v2026.7.7.2）。超越 v1 的方式：

1. **新增子系统的覆盖**：Kanban Worker Lanes、Trust Engine、观测性插件等 v1 不存在的内容
2. **更准确的数字**：v1 有些数字因快速迭代已过时
3. **更完善的章节结构**：14 章 vs v1 的 12 章，单独成章的 Kanban、交互界面
4. **使用指南部分**：v1 几乎没有使用指南，v2 每章必须有（章节模板要求）

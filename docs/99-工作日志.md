## 方法论

### 分析方法

1. **逐文件实读，不靠猜测** — 每个结论必须基于实际读过的代码，不凭模块名或文件名推断内容
2. **分层验证**
   - 结构层：用 glob/ls 确认文件和目录确实存在
   - 代码层：用 grep 验证函数、类、变量确实存在且被使用
   - 关系层：用 grep 追踪 import 链和调用链，不靠推理画关系图
3. **引用溯源** — 文档中的每个关键断言标注源文件和行号
4. **不确定就标注** — 对动态加载、运行时行为等无法静态确认的逻辑，标注"待确认"

### 工作流程

采用 **"分析 + 审核" 双 Agent 流程**：

```
分析 Agent (Explore/sonnet) → 产出草稿 → 审核 Agent (general/sonnet) → 独立验证 → 主线 (opus) 合并 → 人工审阅
```

- **分析 Agent**：读源码，写文档草稿，标注行号引用
- **审核 Agent**：独立读源码，逐条验证分析 Agent 的断言，产出审核报告
- **主线 (opus)**：合并结果，标记差异，交付人工审阅
- **人工审阅**：用户确认无误后才进入下一阶段

### 防幻觉策略

| 策略 | 说明 |
|------|------|
| 逐文件实读 | 不凭文件名推断内容 |
| 分层验证 | 结构层 → 代码层 → 关系层 |
| 引用溯源 | 关键断言标注文件:行号 |
| 双 Agent 交叉验证 | 分析和审核独立读码 |
| 分批交付 | 每阶段交付审阅，不一次性生成 |
| 不确定标注 | 无法确认的标"待确认" |
| 控制单次范围 | 每次聚焦一个子系统 |

### 审核 Agent 职责

1. **存在性验证** — 文件路径、类名、函数名是否存在
2. **关系验证** — 调用关系、import 链是否成立
3. **逻辑验证** — 代码行为描述是否与源码一致
4. **遗漏检查** — 是否有文件被忽略
5. **产出审核报告** — 每条标记 ✅ 已验证 / ❌ 错误 / ⚠️ 存疑

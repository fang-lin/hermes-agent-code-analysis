# 工作流程手册

本文档是跨会话的执行指南。每次新对话开始时，主线 Agent 必须先读本文件和 `PROGRESS.md`，确认当前进度后再继续工作。

准备阶段 → 逐章生产 → 阶段性完整性审核 → 英文翻译 → 终审

## 二、每次对话的启动清单

新对话开始后，主线 Agent 必须：

```
1. 读 CLAUDE.md          — 了解项目规则
2. 读 docs/WORKFLOW.md   — 了解工作流程（本文件）
3. 读 docs/PROGRESS.md   — 了解当前进度
4. 读 docs/zh/99-工作日志.md 的最后一节 — 了解上次做到哪、遇到了什么问题
5. 确认当前章节和步骤，告知用户："上次完成了第 X 章，本次继续第 Y 章"
6. 等用户确认后开始工作
```

**禁止在没读 PROGRESS.md 的情况下开始任何写作工作。**

## 三、单章生产流程

每一章的生产严格按以下步骤执行。步骤编号用于 PROGRESS.md 中记录进度。

### 步骤 1：准备（主线 opus）

1. 确认章节范围：读 CLAUDE.md 中的章节定义，明确本章覆盖哪些模块/文件
2. 拉出本章涉及的源码文件清单（用 glob/grep）
3. 拉出本章对应的官方文档清单（website/docs/ 中的相关文件）
4. 将清单记录到工作日志

### 步骤 2：分析 Agent 产出草稿（sonnet）

1. 启动分析 Agent (sonnet)，传入：
   - 章节模板（docs/CHAPTER_TEMPLATE.md）
   - 本章涉及的源码文件路径清单
   - 对应的官方文档路径清单
   - 明确指令：按模板结构写作，使用指南部分参考官方文档但用叙事风格重写
2. 分析 Agent 产出草稿
3. **立即将草稿保存到文件**（docs/zh/drafts/XX-章节名-draft.md），不要只存在上下文中

### 步骤 3：并行审核（两个 sonnet Agent，互不可见）

1. 读取 `prompts/factual-review.md`，启动事实审核 Agent
2. 读取 `prompts/literary-review.md`，启动文学性审核 Agent
3. 两个 Agent 并行运行，各自独立产出审核报告
4. **立即将审核报告保存到文件**：
   - `docs/zh/reviews/XX-factual-review.md`
   - `docs/zh/reviews/XX-literary-review.md`

### 步骤 4：二次验证（主线 opus）

1. 读取事实审核报告中所有 ⚠️ 和 ❌ 项
2. 对每一项，自己独立读源码验证
3. 记录验证结果：确认/推翻审核 Agent 的判断
4. 如果 opus 验证结果和审核 Agent 不一致，以 opus 实际读码结果为准

### 步骤 5：写入正式文档（主线 opus）

1. 基于草稿，整合以下修改：
   - 事实审核确认的 ❌ 项：修正
   - 事实审核确认的 ⚠️ 项：修正或标注"待确认"
   - 文学性审核的改进建议：采纳合理的
2. 写入正式文件：`docs/zh/XX-章节标题.md`
3. commit hash 填入文档头部

### 步骤 6：增量复核（sonnet）

1. 对步骤 5 中因文学性建议而修改的段落，启动事实审核 Agent 做增量复核
2. 确保文学性修改没有引入新的事实错误

### 步骤 7：更新元数据（主线 opus）

1. 更新 `docs/zh/98-审核报告汇总.md`：本章审核统计
2. 更新 `docs/PROGRESS.md`：标记本章完成
3. 更新 `docs/zh/99-工作日志.md`：记录本章的关键信息
4. 清理草稿和审核报告（可选保留，看用户意愿）

## 四、阶段性完整性审核

在以下节点启动完整性审核：

| 节点 | 触发条件 | 检查范围 |
|------|----------|----------|
| 阶段 1 | 00-01 完成 | 第一部分 |
| 阶段 2 | 02-05 完成 | 第一、二部分 |
| 阶段 3 | 06-09 完成 | 第一、二、三部分 |
| 终审 | 10-13 完成 | 全部 14 章 |

完整性审核流程：
1. 读取 `prompts/completeness-review.md`，启动完整性审核 Agent
2. 审核 Agent 检查 5 项全局指标
3. 主线根据审核结果，回补遗漏内容
4. 记录到工作日志

## 五、跨会话续接规则

### 一章未完成时会话中断

如果某章做到一半会话结束了：

1. PROGRESS.md 中记录到了哪个步骤（如"第 03 章，步骤 3 完成，步骤 4 未开始"）
2. 下次对话续接时：
   - 读取已保存的草稿文件和审核报告
   - 从中断的步骤继续，不要重新开始

### 一章已完成时

1. PROGRESS.md 中标记该章完成
2. 下次对话直接开始下一章

### 用户要求修改已完成章节

1. 修改内容后，需要重新走步骤 3-6（审核流程）
2. 更新 PROGRESS.md 标注"已修订"

## 六、质量红线

以下情况必须停下来等用户确认，不得自行决定：

1. **审核 Agent 发现 3 个以上 ❌**：可能是分析 Agent 的理解方向有误
2. **二次验证推翻审核 Agent 超过 50% 的发现**：可能是审核 prompt 需要调整
3. **完整性审核发现某章遗漏了一个完整子系统**：可能需要调整章节结构
4. **源码结构与 CLAUDE.md 中的章节定义不匹配**：可能代码又更新了

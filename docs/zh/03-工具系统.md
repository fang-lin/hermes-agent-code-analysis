## Agent 只会思考，工具让它能做事

在前两章中，我们看到 AIAgent 的核心循环是"调用模型 → 解析响应 → 执行工具 → 循环"。模型负责决策（"我应该搜索一下网页"），但真正去搜索的不是模型——而是工具系统。没有工具，Agent 就只是一个会聊天的 API 客户端。

如果你想理解 Agent 的执行能力边界，或者遇到"工具没被调用"、"工具结果不对"、"命令被拒绝"等问题需要排查，这一章是入口。

## 使用指南

### 基本用法

工具对用户是透明的——你不需要手动调用工具，模型会根据需要自动选择。但以下几个操作影响工具的可用性：

```bash
hermes tools              # 交互式管理工具集启用/禁用
hermes tools enable mcp   # 启用 MCP 工具集
hermes tools disable browser  # 禁用浏览器工具
```

### 配置

```yaml
# config.yaml 中与工具系统相关的配置
toolsets: ["hermes-cli"]         # 当前使用的工具集

platform_toolsets:
  telegram: ["hermes-telegram"]  # 平台级工具集覆盖

approvals:                       # 命令审批（顶层键）
  mode: "smart"                  # manual/smart/off
  cron_mode: "deny"              # cron 任务遇危险命令：deny/approve

security:
  tirith_fail_open: true         # Tirith 不可用时放行（默认）

tool_search:                     # 渐进式工具披露（详见 MCP 一节）
  enabled: auto                  # 默认 auto：可延迟工具的 schema 占用超上下文 10% 才自动激活

mcp_servers:                     # MCP 服务器配置（顶层平铺键，非嵌套 mcp.servers）
  filesystem:
    command: "npx"
    args: ["-y", "@anthropic/mcp-server-fs"]
```

### 常见场景

**场景一：接入 MCP 工具。** 在 `config.yaml` 的 `mcp_servers` 中添加服务器配置，重启后 `hermes tools` 就能看到新的 `mcp-*` 前缀工具集。MCP 工具和内置工具使用完全相同的注册接口，从 Agent 角度没有区别。

**场景二：给 Telegram 禁用浏览器。** 在 `config.yaml` 的 `platform_toolsets.telegram` 中配置不包含 browser 系列的工具集——或者用 `hermes tools` 在 Telegram 平台页面下禁用浏览器类。

**场景三：开启 smart 审批。** 设置 `approvals.mode: "smart"`，辅助 LLM 会自动判断命令是否安全（APPROVE/DENY/ESCALATE，`approval.py:1960-1972` 的提示词原文），只有不确定的才打扰用户。在 Gateway 场景下特别有用。

### 排错指引

| 问题 | 排查方向 |
|------|---------|
| 模型不调用某个工具 | `hermes tools` 确认工具已启用；检查 `check_fn` 是否返回 True（30 秒 TTL 缓存，`registry.py:134`）；确认工具在当前平台 toolset 中；开了 `tool_search` 的话非核心工具要先被 `tool_search` 检索到才可见 |
| 工具注册了但从未执行 | 检查 `pre_tool_call` 钩子是否拦截了（`model_tools.py:1175`）；检查 `_AGENT_LOOP_TOOLS` 保留名单（`model_tools.py:600`） |
| 工具调用报错 | 检查 `agent.log`（工具错误经过 `_sanitize_tool_error` 净化，`model_tools.py:630`，日志中有原始异常）；`handle_function_call()`（`model_tools.py:1019`）是调度入口 |
| 命令被拒绝 BLOCKED | 可能原因：HARDLINE 匹配（无法绕过，`approval.py:366`）、sudo stdin guard（`sudo -S` 猜密码，`:448`）、DANGEROUS 匹配（`:547`）、Gateway 审批超时 300 秒。查看 `config.yaml` 的 `command_allowlist` |
| MCP 工具没出现 | 检查 `mcp_servers` 配置是否正确；MCP 服务进程是否启动成功（查看 `agent.log` 中的 MCP 连接日志）|
| 工具结果被截断 | 单个结果超 100K chars（`budget_config.py:17`）会被持久化到 `/tmp/hermes-results/`，消息中只保留 1,500 chars 预览 + 路径；模型可以用 `read_file` 读完整内容 |
| 工具执行超时 | 终端命令默认 180 秒超时（`terminal.timeout` 可调） |
| 记忆/技能写入被挂起 | v0.17 起有独立的**写入审批**（`tools/write_approval.py`）：开启 `write_approval` 后，记忆和技能的持久化写入要经用户批准——尤其针对后台自改进 review 的自主写入 |

## 架构与实现

### 一个工具长什么样？

理解工具系统之前，先看一个具体的工具定义。所有 69 个工具都遵循同一个三步模式。以 `read_file` 为例（`tools/file_tools.py`）：

**❶ 定义 Schema** — 告诉模型"这个工具叫什么、接收什么参数"，使用 OpenAI function-calling 的标准 JSON 格式：

```json
{
  "name": "read_file",
  "description": "Read a text file with line numbers...",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {"type": "string"},
      "offset": {"type": "integer", "default": 1},
      "limit": {"type": "integer", "default": 500}
    },
    "required": ["path"]
  }
}
```

**❷ 实现 Handler** — 一个普通的 Python 函数，接收 `args` 字典和关键字参数，返回 JSON 字符串：

```python
def _handle_read_file(args, **kw):
    tid = kw.get("task_id") or "default"
    return read_file_tool(path=args.get("path", ""), ...)
```

返回值必须是 JSON 字符串。`registry.py` 提供两个辅助函数：`tool_result(data)`（`registry.py:754`）返回成功结果，`tool_error(message)`（`registry.py:740`）返回错误。

**❸ 注册** — 在模块级别调用 `registry.register()`：

```python
registry.register(
    name="read_file", toolset="file",
    schema=READ_FILE_SCHEMA, handler=_handle_read_file,
    check_fn=_check_file_reqs, emoji="📖",
    max_result_size_chars=100_000
)
```

`check_fn` 是可选的可用性检查函数，在每次获取工具定义时调用（结果缓存 30 秒，`_CHECK_FN_TTL_SECONDS`，`registry.py:134`；还有一个防抖动细节：刚刚成功过的检查遇到瞬时失败会先按"上次的 True"处理一小段时间，避免一次网络闪断让工具集忽隐忽现）。以 Browser 工具为例，它的 check_fn 检查 Playwright 是否已安装——若返回 False，该工具被静默移出可用列表，模型不会看到它的 schema。这就是为什么某些工具"存在但不可用"——它们注册了但 check_fn 把它们过滤掉了。

为什么选择 OpenAI 的 function-calling 格式？因为这是行业事实标准——所有主流 LLM 都理解。Transport 层在必要时将其转换为各 Provider 的原生格式（以 Anthropic 的 `input_schema` 为例），但**工具定义只写一次**。

### 工具怎么被发现和加载的？

93 个顶层工具文件不是在启动时全部 import 的——那会拉入 Playwright、faster-whisper 等重依赖，拖慢启动。`discover_builtin_tools()`（`tools/registry.py:58`）用两阶段加载：

```mermaid
%%{init: {"theme": "neutral", "themeVariables": {"fontSize": "14px"}, "flowchart": {"nodeSpacing": 15, "rankSpacing": 25}}}%%
flowchart LR
    FILES["tools/*.py"] --> AST["AST 扫描<br/>不执行代码"]
    AST -->|有 register 调用| IMPORT["按需 import<br/>触发模块级 register"]
    AST -->|无| SKIP["跳过"]
    IMPORT --> REG["ToolRegistry<br/>69 个工具"]
```

**图：两阶段工具发现——AST 扫描确认必要性，再按需 import**

为什么用 AST 而非"import 所有文件然后 try-except"？因为 import 会执行模块级代码，可能有副作用（以建立数据库连接为例）。AST 只解析语法树不执行代码，是零副作用的发现机制。两条失败路径的可观测性不同：AST 解析失败（文件语法错误）时 `_module_registers_tools` 直接返回 False——**完全静默**，连日志都没有（`registry.py:50-51` 的 except 分支）；通过扫描但 import 阶段抛异常才会记 warning（`registry.py:74`）。排查"新工具文件没被加载"时要意识到前一种情况在日志里查不到。

### 从 LLM 返回 tool_call 到结果回填

在 02 章的核心循环中，当 LLM 返回包含 `tool_calls` 的响应时，控制权转到工具层。完整的执行路径是：

```mermaid
%%{init: {"theme": "neutral", "themeVariables": {"fontSize": "14px"}, "flowchart": {"nodeSpacing": 15, "rankSpacing": 25}}}%%
flowchart TD
    TC["LLM 返回 tool_calls"] --> PAR{{"并行判断"}}
    PAR -->|PARALLEL_SAFE| CONC["ThreadPoolExecutor 并行"]
    PAR -->|PATH_SCOPED 路径不重叠| CONC
    PAR -->|其他| SEQ["串行"]
    CONC --> COERCE["coerce_tool_args 参数类型转换"]
    SEQ --> COERCE
    COERCE --> PRE["pre_tool_call 钩子（可拦截）"]
    PRE --> APPROVE["安全检查"]
    APPROVE --> EXEC["执行 handler"]
    EXEC --> POST["post_tool_call 钩子（观测）"]
    POST --> TRANSFORM["transform_tool_result 钩子（可修改）"]
    TRANSFORM --> SIZE["结果大小治理（handle_function_call 返回后）"]
    SIZE --> BACK["结果回填 messages"]
```

**图：工具调用的完整执行路径——包含参数转换、三个钩子、安全检查、结果治理**

关键步骤说明：

1. **并行判断**（`agent/tool_dispatch_helpers.py:104`）：`_should_parallelize_tool_batch()` 按三档分诊：
   - `_NEVER_PARALLEL_TOOLS`（`:42`）：`clarify` 一票否决整批并行
   - `_PATH_SCOPED_TOOLS`（`:60`）：`read_file`、`write_file`、`patch`——目标路径互不重叠才并行，重叠回退串行。**注意 `read_file` 走这一档**：它虽然也出现在安全名单里，但路径检查分支在前，"只读所以随便并行"不成立
   - `_PARALLEL_SAFE_TOOLS`（`:45`，11 个条目）：`web_search`、`search_files`、`session_search`、`vision_analyze` 等不落盘的只读工具，无条件放行
   - MCP 工具：由 MCP server 的并行声明控制（`_is_mcp_tool_parallel_safe`，`:91`）
   - 如果参数解析失败（JSON 非法），直接回退串行

2. **参数类型转换**（`coerce_tool_args()`，`model_tools.py:650`）：对参数做类型强制转换（以字符串 `"42"` 转为整数 `42` 为例）。排查"参数类型错误"时从这里入手

3. **三个钩子依次执行**：
   - `pre_tool_call`（`model_tools.py:1175`）：执行前触发，插件可返回 block 消息**阻止执行**——排查"工具注册了但从未执行"时，检查是否被 pre_tool_call 拦截。调用方还可以用 `skip_pre_tool_call_hook` 参数（`:1029`）显式跳过
   - `post_tool_call`（发射函数 `_emit_post_tool_call_hook`，`model_tools.py:968`，在成功/异常两条路径 `:1194/:1294` 各发射一次）：执行后触发，仅观测（返回值被忽略），用于记录耗时等指标
   - `transform_tool_result`（`model_tools.py:1309`）：最后触发，插件可返回字符串**替换结果**

4. **安全检查**：terminal 工具的命令经过 `check_all_command_guards()`（详见下文"安全防线"）

5. **结果大小治理**：注意——治理不在 `handle_function_call()` 内部，而在上层的 `agent/tool_executor.py`（1,646 行，管预算、持久化、中断处理），在 handler 返回之后执行。详见下文"结果大小治理"

6. **回填**：最终结果作为 `{"role": "tool", "content": result}` 消息追加到 `messages` 列表

注意：工具错误信息会经过 `_sanitize_tool_error()`（`model_tools.py:630`）净化——防止异常中的 framing tokens 进入模型上下文。这意味着 debug 时工具返回值中的错误信息可能与原始异常不同，需要查 `agent.log` 获取完整异常。

### 工具集：同一批工具，不同场景

并非所有场景都需要所有工具。CLI 用户可能需要浏览器，但 Telegram 群聊不需要——想象一个 Agent 在群里自动打开浏览器访问网页。

`toolsets.py` 定义了一个**共享核心 + 平台扩展**的模型。`_HERMES_CORE_TOOLS`（`toolsets.py:31`）列出 49 个核心工具，所有平台共享。每个平台的 toolset 在核心基础上增减：

工具集支持递归组合——`resolve_toolset()`（`toolsets.py:687`）做 DFS 展开并检测环路。用户可以通过 `config.yaml` 的 `platform_toolsets` 覆盖任何平台的工具集。如果某个工具的依赖不满足（以 Playwright 未安装为例），`check_fn` 在注册时返回 False，该工具被静默从可用列表中移除。

顺带一提 v0.18 的一个减法：v0.14 时代的 `mixture_of_agents` 模型工具已被整体移除——多模型聚合重构成了 Agent 循环层的 MoA 模式（第 02 章），不再占用工具位。

### 安全：不止三道防线

工具系统最敏感的部分是安全——Agent 执行的命令可能删除文件、修改系统配置、发起网络请求。Hermes 在工具调用路径上的防线各守不同的攻击向量：**命令审批**拦截危险的命令意图（用正则识别 `rm -rf /` 这样的操作）；**路径 + URL 安全**拦截结构性绕过攻击（路径穿越和 SSRF）；**Tirith** 拦截内容伪装攻击（用语义分析发现正则看不见的威胁）。v0.17-0.18 又加了两个面：**写入审批**管住 Agent 对自身记忆/技能的自主修改，**威胁模式库**统一各处上下文扫描的提示注入检测。

#### 第一道：命令审批（approval.py，3,242 行）

approval.py 从 v0.14 的 1,424 行翻倍到 3,242 行——增量主要在上下文感知（cron/会话状态）和 Gateway 异步审批。`check_all_command_guards()`（`approval.py:2537`）是入口，按优先级执行六级检查：

```mermaid
%%{init: {"theme": "neutral", "themeVariables": {"fontSize": "14px"}, "flowchart": {"nodeSpacing": 15, "rankSpacing": 25}}}%%
flowchart TD
    CMD["命令进入"] --> CONTAINER{{"容器化？"}}
    CONTAINER -->|是| ALLOW1["放行"]
    CONTAINER -->|否| HARD{{"HARDLINE?"}}
    HARD -->|匹配| DENY["无条件拒绝"]
    HARD -->|否| YOLO{{"--yolo?"}}
    YOLO -->|是| ALLOW2["放行"]
    YOLO -->|否| TIRITH["Tirith 扫描"]
    TIRITH --> DANGER{{"DANGEROUS?"}}
    DANGER -->|匹配| APPROVAL["需要审批"]
    DANGER -->|不匹配| ALLOW3["放行"]
```

**图：命令审批的六级检查——从容器豁免到硬核封锁到 DANGEROUS 模式匹配**

检查流程逐级说明：

1. **容器化环境检测**——Docker、Singularity、Modal、Daytona 内的命令直接放行，因为容器本身就是沙箱
2. **HARDLINE_PATTERNS**（`approval.py:366`）——无条件拒绝的命令模式，即使 `--yolo` 也不能绕过。共 12 个模式：`rm -rf /`、`mkfs`、fork bomb 等极端危险操作。这是 Hermes 的安全底线
2b. **sudo stdin guard**（模式定义 `approval.py:448`）——阻止 Agent 通过 `sudo -S` 管道猜密码攻击，同样无法被 `--yolo` 绕过。错误信息为 `"BLOCKED: sudo password guessing via stdin"`
3. **--yolo / approvals.mode=off**——用户显式选择跳过所有审批（除 HARDLINE 和 sudo guard 外）。用户自定义的 deny 规则（`approvals` 配置里的 fnmatch glob）在 yolo 之前生效——它是"代码内置硬底线"的用户可编辑版
4. **Tirith 内容扫描**（`tools/tirith_security.py`，871 行）——外部 Rust 二进制，检测 Unicode 同质字形伪装 URL 等正则无法捕获的内容级威胁。默认 `fail_open=true`（不可用时放行）。*（Tirith 的完整介绍见下文"第三道防线"一节）*
5. **DANGEROUS_PATTERNS**（`approval.py:547-762`）——**73 个**正则匹配危险命令模式（`rm -rf`、`chmod 777`、`DROP TABLE`、`git reset --hard`、`curl | sh`、killall 强杀系列等；v0.14→v0.18 从 52 条扩到 73 条，新增的以 Windows PowerShell 破坏命令为代表）
6. **审批**——匹配到 DANGEROUS 或 smart 模式判定为可疑时，触发审批。审批方式取决于平台：CLI 是交互式提示（once/session/always/deny），Gateway 是异步等待用户响应（最长 300 秒），smart 模式（`approvals.mode` 三值之一，`:1797`）用辅助 LLM 自动判断（APPROVE/DENY/ESCALATE）

审批结果持久化：`always` 级别写入 `config.yaml` 的 `command_allowlist`，后续同类命令不再询问。

Gateway 场景下审批等待 300 秒后超时，返回 `"BLOCKED: Command timed out without user response"`——这是 Gateway 下命令"被拒绝"最常见的原因之一。Cron 模式下有独立的配置（`approvals.cron_mode`）：`deny` 时危险命令直接拒绝（默认，Agent 会找别的办法），`approve` 时全部放行。

#### 第二道：路径和 URL 安全

**路径安全**（`tools/path_security.py`，43 行）——防止路径遍历攻击。`validate_within_dir()` 用 `path.resolve().relative_to(root)` 确保文件操作不逃逸出允许目录。

**URL 安全**（`tools/url_safety.py`，503 行）——防止 SSRF（Server-Side Request Forgery）。`is_safe_url()` 对 URL 做 DNS 解析后检查 IP，**fail-closed**（DNS 失败则拒绝）。永久封锁所有云平台元数据端点：`169.254.169.254`（AWS/GCP）、`metadata.google.internal`、Azure/DigitalOcean 等。即使配置了 `security.allow_private_urls: true` 放开私网 IP，元数据端点仍不可访问——这防止了 Agent 在云服务器上读取 VM 的 IAM 凭证。

#### 第三道：Tirith 内容级扫描

Tirith（`tools/tirith_security.py`，871 行）是外部 Rust 二进制，检测正则无法捕获的内容级威胁。首次使用时从 GitHub releases 自动下载，验证 SHA-256 校验和和 cosign 供应链签名。下载在后台线程进行，不阻塞 Agent 启动。

#### 第四道（v0.17 新增）：写入审批——管住 Agent 改自己

命令审批管的是 Agent **对外**做什么；`tools/write_approval.py`（493 行）管的是 Agent **对自己**做什么。Agent 有两个跨会话存活的持久存储：记忆（MEMORY.md/USER.md，约 200 字符的小条目）和技能（SKILL.md + 附件，可达 10-100 KB）。写入来自两个源头：前台（用户在场的正常轮次）和 **background_review**（每轮结束后自主决定"要记住什么"的自改进 fork——模块注释直言这是用户抱怨"记了错误假设"的来源）。开启 `write_approval` 后，可按子系统（记忆/技能）分别要求写入先经用户批准，待批条目进入 pending 存储。

命令审批保护你的系统，写入审批保护 Agent 自己的长期状态不被它自己污染——这是"自改进 Agent"特有的安全面。

#### 统一威胁模式库

`tools/threat_patterns.py`（284 行）是 v0.18 的一次防线整备：提示注入/promptware/数据外渗的检测模式原来散在各处，现在收拢为单一真相源。实际接入的消费方（按 import 关系数）是三个：`agent/prompt_builder.py`（上下文组装扫描）、`tools/memory_tool.py`（记忆写入扫描）、`tools/cronjob_tools.py`（cron 提示注入扫描，用其中的隐形字符名单）——模式改一处，三处同时生效。（模块自己的 docstring 还提到工具结果定界系统，但那套机制——`<untrusted_tool_result>` 包裹与定界符中和——在 `tool_dispatch_helpers.py` 里自成一体，并未 import 本模式库；docstring 与实际引用已轻微脱节。）

### MCP：让外部工具成为一等公民

通过 MCP（Model Context Protocol），任何实现了 MCP 协议的外部服务都能注册为 Hermes 的工具。

关键设计：**MCP 工具和内置工具使用完全相同的注册接口**。`discover_mcp_tools()`（`tools/mcp_tool.py`，5,627 行）在启动时连接配置的 MCP servers，获取工具列表，然后用同一个 `registry.register()` 注册进 ToolRegistry。从 Agent 角度看，MCP 工具和内置工具没有区别。

MCP 工具的 toolset 以 `mcp-` 开头（以 `mcp-filesystem` 为例），便于按需启用禁用。注册时有重名保护：MCP 工具之间可互相覆盖，但 MCP 工具不能覆盖内置工具（反之亦然）。当 MCP server 的工具列表变化时，server 发送 `tools/list_changed` 通知，Hermes 自动做 deregister + 重新注册，无需重启 Agent。

MCP 的实现细节值得了解：

- **独立后台事件循环**（`mcp_tool.py:69-76`）：MCP 运行在 daemon 线程中的专用 asyncio 事件循环上，工具调用跨线程调度。这解释了为什么 MCP 工具的超时行为和内置工具不同
- **三种传输协议**（`mcp_tool.py:5`）：stdio（子进程通信，最常用）、HTTP/StreamableHTTP（远程服务）、SSE。配置中的 `command` 字段表示 stdio，`url` 字段表示 HTTP/SSE
- **Sampling 功能**（`mcp_tool.py:71`）：MCP server 可以通过 `sampling/createMessage` 反向请求 LLM 完成——这意味着 MCP 工具调用可能触发嵌套的 LLM 请求，排查意外的 LLM 调用时需要考虑这个路径
- **自动重连**：连接失败时指数退避重试。如果 MCP server 持续不可用，该 server 的工具被静默移除
- **安全**：MCP 子进程只继承白名单环境变量（PATH、HOME、USER 等），敏感凭证不泄露到外部进程
- **孤儿看门狗**（`tools/mcp_stdio_watchdog.py`，184 行，v0.18 新增）：stdio MCP 服务器是 Hermes 的直接子进程，Hermes 被 `kill -9` 时正常清理路径来不及跑，会留下孤儿 `npx`/node 进程。看门狗作为独立监督进程盯着父进程死亡，代为收尸

#### 工具太多怎么办：渐进式披露

MCP 接得越多，工具 schema 越大——每个工具的定义都占模型上下文。`tools/tool_search.py`（735 行，v0.18 新增）给出的答案是**渐进式工具披露**。注意它的默认档位是 `auto`（`ToolSearchConfig.from_raw`，`tool_search.py:78-83`）：配置里完全不写这个键时，一旦可延迟工具的 schema 占用达到上下文窗口的 10%（窗口未知则 20K token 门槛）就**自动激活**——不是"默认关闭需手动开"。机制：MCP 和非核心插件工具不再直接进入模型可见的 tools 数组，而是被三个桥接工具替代——`tool_search`（按需检索）、`tool_describe`（查看完整定义）、`tool_call`（间接调用）。模型需要什么就搜什么，用到才展开 schema。核心 Hermes 工具永不延迟——高频工具的直接可见性不牺牲。

### 结果大小治理

不管工具是内置的还是通过 MCP 接入的，它们都面临同一个现实问题：输出可能很大。如果工具返回了 1MB 的文件内容，全部塞进消息历史会怎样？上下文爆炸、费用飙升、延迟变高。Hermes 用三层机制控制：

```mermaid
%%{init: {"theme": "neutral", "themeVariables": {"fontSize": "14px"}, "flowchart": {"nodeSpacing": 15, "rankSpacing": 25}}}%%
flowchart TD
    L1["❶ 工具内部预截断<br/>终端 ≤ 50K chars · 文件 ≤ 2000 行"]
    L2{{"单结果 > 100K chars？"}}
    L2Y["持久化到 /tmp/hermes-results/<br/>消息中替换为预览 + 路径"]
    L2N["保持原样"]
    L3{{"一轮总量超预算？"}}
    L3Y["按大小降序持久化最大结果"]
    L1 --> L2
    L2 -->|是| L2Y --> L3
    L2 -->|否| L2N --> L3
    L3 -->|是| L3Y
```

**图：工具结果大小治理——❶ 预截断 → ❷ per-result 持久化 → ❸ per-turn 预算**

阈值都在 `tools/budget_config.py`（114 行）：单结果阈值默认 100K chars（`DEFAULT_RESULT_SIZE_CHARS`，`:17`），预览 1,500 chars（`:19`）。**每轮总预算 v0.18 起是动态的**：取模型上下文窗口的 30%（`_PER_TURN_WINDOW_FRACTION`，`:76`），下限 16K、上限 200K（`:81/:18`）——小窗口模型自动收紧，大窗口模型不至于被 200K 死数卡住。

这个设计的巧妙之处：**不丢弃信息**。超大结果被持久化到文件系统，消息中放 1,500 chars 预览 + 路径，模型需要完整内容时用 `read_file` 读取。代价是多一次工具调用，但不会无声地丢失信息。

特殊处理：`read_file` 虽然注册时设置了 `max_result_size_chars=100_000`，但 `budget_config.py:11` 的 `PINNED_THRESHOLDS` 在运行时将其覆盖为 `float('inf')`——因为如果 read_file 结果也被持久化，模型就需要用 read_file 读 read_file 的持久化结果，陷入无限循环。这个覆盖机制通过 `BudgetConfig.resolve_threshold()`（`:37`）实现，优先级链为：pinned → tool_overrides → registry → default。

### 后台委托：让子 Agent 异步跑

第 02 章讲的 `delegate_task` 是同步的——父 Agent 等所有子任务完成才继续。v0.18 新增 `delegate_task(background=true)`：子 Agent 被派到模块级 daemon 执行器上异步跑（`tools/async_delegation.py`，531 行 + `tools/daemon_pool.py`，64 行），父 Agent 立即拿到一个句柄，继续做别的事，之后凭句柄查询状态/收结果。这把"派活等结果"变成了"派活先走、回头收账"——适合研究扇出这类长耗时子任务。与 02 章的心跳监控、汇总预算机制共用同一套底座。

### Computer Use：桌面控制

除了浏览器工具（`browser_tool.py`，4,803 行，控制浏览器页面），hermes-agent 还有一个**桌面级**的自动化工具——`computer_use`（`tools/computer_use/` 子目录，8 个 .py 文件，v0.18 新增 `doctor.py` 自诊断和 `permissions.py` 权限检查）。

Computer Use 基于 cua-driver（一个独立安装的 MCP stdio 二进制，支持 macOS/Windows/Linux 三平台——`check_computer_use_requirements()`，`tool.py:899`；Linux 目前走 X11/XWayland），让 Agent 能控制整个桌面——点击按钮、输入文字、截屏分析。和浏览器工具的关键区别：浏览器工具只能操作浏览器页面，Computer Use 能操作任何 GUI 应用（以 Finder、系统偏好设置为例）。

一个重要的设计选择：`computer_use` 的 schema 使用 OpenAI function-calling 格式（`tools/computer_use/schema.py`），而非 Anthropic 的原生 Computer Use 格式——这意味着**所有支持工具调用的模型**都能驱动它，不限于 Claude。

通过 `hermes tools` 安装 cua-driver，配置后在 `config.yaml` 中启用 `computer_use` 工具集。

### 终端后端：7 种执行环境

Agent 的 `terminal` 工具不一定在本地执行命令。`tools/environments/` 提供了 7 种后端：

| 后端 | 文件 | 场景 |
|------|------|------|
| Local | `local.py` | 默认，直接在宿主机执行 |
| Docker | `docker.py` | 容器隔离，适合不信任的命令 |
| SSH | `ssh.py` | 远程服务器执行 |
| Daytona | `daytona.py` | 云沙箱，空闲时休眠 |
| Singularity | `singularity.py` | HPC 容器 |
| Modal (SDK) | `modal.py` | Serverless，按用量计费 |
| Modal (托管) | `managed_modal.py` | 通过工具网关代理 |

（v0.14 时代还有 Vercel Sandbox 后端，已随 Vercel AI Gateway 一并移除。）

7 种后端的核心差异在于**隔离级别和执行位置**：Local 无隔离，命令直接运行在宿主机；Docker/Singularity 提供进程级容器隔离，适合运行不信任的代码；Modal 是完全无状态的云沙箱，启动延迟高但无运维负担；Daytona 和 SSH 则针对持久化远程环境，SSH 用于已有的服务器，Daytona 提供生命周期管理（空闲时自动休眠以节省费用）。

选择后端时，三个维度最重要：**安全需求**（是否需要沙箱隔离）、**持久状态需求**（是否需要跨任务保留文件系统）、**执行位置**（本地、私有服务器还是云端）。

所有后端实现 `BaseEnvironment` 接口（`tools/environments/base.py`，959 行），Agent 和工具代码不需要关心命令在哪里执行——切换后端只需改 `config.yaml` 的 `terminal.backend`。但某些后端有额外依赖（以 Modal 为例，需要 `pip install modal` 和 Modal 账号配置）。

排查终端后端问题时需要了解两个隐性行为：

- **CWD 持久化**（`base.py:282`）：Session 通过在命令输出中嵌入 `__HERMES_CWD_{session_id}__` 标记来传递工作目录。远端后端（SSH、Modal）解析这个标记来同步 CWD。如果命令输出恰好包含类似字符串，可能导致 CWD 解析错误
- **Snapshot 退化**（`base.py:443`）：如果 `init_session()` 失败（`snapshot_ready=False`），后续每个命令都退化为 `bash -l` 执行，绕过了环境变量继承——之前命令设置的环境变量在后续命令中消失。表现为"命令结果不对"
- **后台进程挂起**（`base.py:568-574`，issue #8340）：命令后台化时（`cmd &`），grandchild 进程继承 stdout pipe 的写端，bash 退出后管道仍开着，drain 线程等不到 EOF，工具一直挂着。设置 `HERMES_DEBUG_INTERRUPT=1` 环境变量可开启详细的 poll 日志帮助定位

### 设计决策

#### 自注册 vs 集中式注册

Hermes 选择了自注册模式——每个工具文件在被 import 时自己调用 `registry.register()`，而不是有一个中央文件逐一列出所有工具。好处：新增工具只需创建文件，不需要修改任何其他文件。代价：工具的注册顺序取决于 import 顺序，但这在实践中不是问题（注册表是字典，按名称查找，不依赖顺序）。

#### MCP 工具和内置工具同接口

MCP 工具可以用完全不同的注册路径——比如单独的 MCP 调度层。但 Hermes 选择让 MCP 工具和内置工具走同一个 `ToolRegistry`，对 Agent 透明。好处：Agent 核心不需要区分工具来源，简化了工具调度逻辑。代价：MCP 工具的错误和内置工具的错误在日志中不容易区分（排查时需要看 toolset 名称的 `mcp-` 前缀）。

#### 结果持久化而非截断

传统做法是直接截断超大工具结果。Hermes 选择持久化到文件 + 消息中放预览。好处：信息不丢失。代价：需要额外一次 `read_file` 工具调用，增加了一轮 Agent 循环。

#### 工具的可见性也是预算

`tool_search` 的渐进式披露是同一哲学在 schema 侧的应用：结果侧"超大内容持久化、按需读取"，定义侧"非核心工具收进检索、按需展开"。两边都是把"全量塞上下文"改成"指针 + 按需取"，代价同样是多一次工具调用。

### 扩展点

1. **新增内置工具**：创建 `tools/<name>.py`，实现 schema + handler + `registry.register()`
2. **新增 MCP 工具**：在 `config.yaml` 的 `mcp_servers` 中添加服务器配置
3. **新增终端后端**：实现 `BaseEnvironment` 接口
4. **自定义工具集**：在 `config.yaml` 的 `toolsets` 或 `platform_toolsets` 中组合
5. **自定义安全策略**：通过 `command_allowlist` 预批准特定命令模式，或 `approvals` 的用户 deny 规则加硬底线

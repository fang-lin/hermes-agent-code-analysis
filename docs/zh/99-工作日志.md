## 方法论

### 分析方法

1. **逐文件实读，不靠猜测** — 每个结论必须基于实际读过的代码，不凭模块名或文件名推断内容
2. **分层验证**
   - 结构层：用 glob/ls 确认文件和目录确实存在
   - 代码层：用 grep 验证函数、类、变量确实存在且被使用
   - 关系层：用 grep 追踪 import 链和调用链，不靠推理画关系图
3. **引用溯源** — 文档中的每个关键断言标注源文件和行号
4. **不确定就标注** — 对动态加载、运行时行为等无法静态确认的逻辑，标注"待确认"

### 工作流程

采用 **"分析 + 审核" 双 Agent 流程**：

```
分析 Agent (Explore/sonnet) → 产出草稿 → 审核 Agent (general/sonnet) → 独立验证 → 主线 (opus) 合并 → 人工审阅
```

- **分析 Agent**：读源码，写文档草稿，标注行号引用
- **审核 Agent**：独立读源码，逐条验证分析 Agent 的断言，产出审核报告
- **主线 (opus)**：合并结果，标记差异，交付人工审阅
- **人工审阅**：用户确认无误后才进入下一阶段

### 防幻觉策略

| 策略 | 说明 |
|------|------|
| 逐文件实读 | 不凭文件名推断内容 |
| 分层验证 | 结构层 → 代码层 → 关系层 |
| 引用溯源 | 关键断言标注文件:行号 |
| 双 Agent 交叉验证 | 分析和审核独立读码 |
| 分批交付 | 每阶段交付审阅，不一次性生成 |
| 不确定标注 | 无法确认的标"待确认" |
| 控制单次范围 | 每次聚焦一个子系统 |

### 审核 Agent 职责

1. **存在性验证** — 文件路径、类名、函数名是否存在
2. **关系验证** — 调用关系、import 链是否成立
3. **逻辑验证** — 代码行为描述是否与源码一致
4. **遗漏检查** — 是否有文件被忽略
5. **产出审核报告** — 每条标记 ✅ 已验证 / ❌ 错误 / ⚠️ 存疑

#### 步骤 13.0：重启分析与升级定调
- **时间**：2026-05-24
- **执行者**：主线 (Gemini-3.1-Pro)
- **事件背景**：
  原分析停留在 v0.11.0 版本。经由用户提醒并执行 `git pull` 后发现，源码库已跃升至 `v2026.5.16`，新增了近 3000 个 Commit。架构引入了 Kanban Worker Lanes、Trust Engine、内置观测性插件等重大变更。
- **架构师决策与教训记录**：
  在本次升级筹备期间，由于系统底层设计将“分析计划”、“草稿”等构思过程自动隔离在不可见的 Artifact 临时沙盒中，遭到了用户的严厉且深刻的批评。用户指出：**“作草稿与沟通过程，计划，决策过程才是项目的本质，代码可以随时生成，但问题分析过程才是项目持久的生命。”**
  这是一个极具架构哲理的指点。因此，我们将所有的升级计划、决策过程和 Agent 编排方法，重新作为项目的一等公民，正式迁回并永久记录在本文档库中。
- **Agent 编排与防幻觉策略 (V2 版)**：
  由于升级跨度极大，为防止主线模型陷入长上下文注意力坍缩和自我确证的幻觉中，我们决定严格沿用并强化原有的多智能体协作：
  1. **主线调度**：负责阅读最新源码，撰写草稿，并记录决策。
  2. **绝对沙盒隔离**：利用底层的 `define_subagent` 派生 `Factual_Reviewer` 与 `Literary_Reviewer`。它们不在当前聊天上下文中，没有主线写草稿的记忆污染。
  3. **强制索证**：子智能体仅凭 `prompts/` 下的模板和物理源码文件对草稿进行交叉盘问。
  4. **仲裁执行**：主线依据子智能体传回的报错，重新验证源码并完成正式文档。
- **下一步行动**：确认该更新基调后，将从 `02-agent核心.md` 等受影响最深的文档开始启动。

# 01 - 一条消息的旅程：Hermes 架构深度解析

## 从一个问题开始

假设你在终端里输入了一句话："帮我搜索最近的 Python 安全漏洞，整理成表格"。Hermes 会怎么处理这条消息？

这不是一个简单的"调用 API 返回文本"的过程。在你按下回车到看到结果之间，这条消息会经历至少五层处理：输入捕获、提示词组装、模型调用、工具执行、结果合成——而且后三步会循环多次。理解这个流程，就理解了 Hermes 的整个架构。

### 请求全景图

```
用户输入
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│ ① CLI / 网关                                                │
│                                                             │
│  CLI: prompt_toolkit → _pending_input 队列 → chat()         │
│  网关: 平台适配器 → MessageEvent → handle_message()          │
│                                                             │
│  共同出口: agent.run_conversation()                          │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│ ② Agent 核心 (run_agent.py)                                 │
│                                                             │
│  准备阶段:                                                   │
│    构建系统提示词 → 预取记忆 → 压缩历史（如需要）             │
│                                                             │
│  ┌─────────────── 核心循环 ───────────────┐                 │

│  │       ▲               │                │                 │

│                                                             │
│  串行 or 并行（按副作用判断）                                 │

```

下面逐站展开。

## 第一站：CLI 捕获你的输入

一切从 `cli.py` 的主循环开始。`HermesCLI` 类用 `prompt_toolkit`（一个 Python 终端 UI 库）构建了一个固定输入区的 REPL。用户输入通过 `prompt_toolkit` 的 TextArea 捕获，放入一个线程安全的队列 `_pending_input`。

主线程的 `process_loop()`（`cli.py:10819`）以 100ms 间隔轮询这个队列：

```python
while not self._should_exit:
    user_input = self._pending_input.get(timeout=0.1)
```

拿到输入后，它做三件事之一：检测是否是拖拽的文件路径、检测是否是斜杠命令（如 `/model`、`/new`），或者——如果是普通消息——调用 `self.chat()`（`cli.py:8404`）。

`chat()` 方法会在一个**后台 daemon 线程**中启动 Agent 调用（`cli.py:8664`），而不是在主线程。为什么？因为主线程需要保持响应——用户随时可能按 Ctrl+C 中断，或者输入新消息覆盖当前任务。如果 Agent 调用阻塞了主线程，这些操作就做不到了。

在后台线程里，`chat()` 调用了 Agent 核心的入口方法（`cli.py:8636`）：

```python
result = self.agent.run_conversation(
    user_message=agent_message,
    conversation_history=self.conversation_history[:-1],
    stream_callback=stream_callback,
)
```

至此，控制权从 CLI 层移交给 Agent 核心。

## 第二站：Agent 核心——一个不停转的循环

`AIAgent.run_conversation()`（`run_agent.py:9627`）是整个系统的心脏。它接收用户消息和历史上下文，返回最终回复。但它做的事情远比"调用一次 API"复杂。

### 循环，而不是单次调用

Agent 的核心是一个 **while 循环**（`run_agent.py:9993`）：

```python
while (api_call_count < self.max_iterations
       and self.iteration_budget.remaining > 0) or self._budget_grace_call:
```

为什么是循环？因为 AI 模型的一次响应可能不是最终答案——它可能说"我需要先搜索一下网页"或"让我读一下这个文件"。这些就是**工具调用**。Agent 循环的逻辑是：

1. 调用模型 → 拿到响应
2. 如果响应包含工具调用 → 执行工具 → 把结果放回消息历史 → 回到第 1 步
3. 如果响应是纯文本（`finish_reason == "stop"`）→ 退出循环，返回最终回复

默认最多循环 90 次（`max_iterations`，`run_agent.py:851`）。但还有第二道限制——`iteration_budget`——这是一个父子 Agent 共享的预算对象（`run_agent.py:951`）。如果你的 Agent 派出了子 Agent，它们共享同一个预算池，防止子 Agent 失控消耗所有迭代次数。

### 在循环开始之前

在进入循环之前，`run_conversation()` 做了几件准备工作：

**系统提示词构建**（`run_agent.py:4543`，`_build_system_prompt()`）。这不是简单的一段固定文本，而是一个多层三明治。源码注释（`run_agent.py:4551-4558`）列出 7 个主要层次：

1. Agent 身份——SOUL.md（如果存在）或内置默认身份
2. 用户/网关提供的额外系统消息
3. 持久化记忆快照——从 MEMORY.md 和 USER.md 读取
4. 技能系统提示——如果加载了技能工具
5. 上下文文件——AGENTS.md、.cursorrules 等项目级指令
6. 当前日期与时间（构建时冻结）
7. 平台格式化提示

实际实现中还有更多子项（帮助引导、工具感知行为引导、订阅提示等），按条件动态追加。

关键设计：这个系统提示**在会话内只构建一次**，之后缓存在 `_cached_system_prompt`（`run_agent.py:9814`）。为什么？因为部分 Provider（如 Anthropic）支持 prefix caching——如果每次请求的系统提示完全相同（byte-identical），服务端可以复用之前的 KV cache，大幅降低延迟和成本。即使 Provider 不支持 prefix caching，稳定的系统提示也有利于本地推理引擎（Ollama、vLLM 等）的 KV cache 复用。如果每次都重新构建（比如插入了变化的时间戳），缓存就失效了。

**记忆预取**（`run_agent.py:9989`）。在第一次 API 调用前，`MemoryManager` 会用用户消息作为查询，预取相关的记忆片段。但这些片段不是注入到系统提示里（那会破坏缓存），而是注入到**用户消息**中（`run_agent.py:10131-10143`）。这就是 Hermes 的"双注入"策略：静态记忆快照走系统提示（稳定前缀），动态检索结果走用户消息（每次不同）。

**上下文压缩**（`run_agent.py:9846-9905`）。如果历史消息已经很长（超过 75% 上下文窗口，阈值定义在 `agent/context_engine.py:59`），在进入循环前就会做最多 3 轮压缩。压缩器（`agent/context_compressor.py`）不是简单截断——它用一个廉价的辅助模型把中间消息摘要化，保护头部 3 条和尾部 6 条消息不被动。摘要预算是被压缩内容的 20%，上限 12,000 token（`context_compressor.py:55-57`）。

## 第三站：跨越 Provider 的鸿沟

循环里每次调用模型，都要面对一个问题：不同 Provider 的 API 格式完全不同。Anthropic 用 `messages` 数组加独立的 `system` 参数；OpenAI 把 system message 塞在 messages 数组里；AWS Bedrock 有自己的 Converse API；OpenAI 的 Codex 用 Responses API。

### 问题的规模

Hermes 不是只支持"几个" Provider。它支持 20+ 个，从云端巨头（OpenAI、Anthropic、Bedrock、Gemini）到开源推理引擎（Ollama、vLLM、llama.cpp）到国内平台（Kimi、MiniMax、通义千问）。如果为每个 Provider 写一套调用逻辑，`run_agent.py` 会膨胀到不可维护的程度。

### Transport 抽象层

```
           AIAgent (run_agent.py)
               │
               │  统一调用接口
               ▼
     ┌──── ProviderTransport ABC ────┐
     │                               │
     │  convert_messages()           │
     │  convert_tools()              │
     │  build_kwargs()        ◄──── 4 个方法抹平差异
     │  normalize_response()         │
     │                               │
     └───┬────────┬────────┬────┬───┘
         │        │        │    │
         ▼        ▼        ▼    ▼
    ┌────────┐┌───────┐┌──────┐┌─────┐
    │ChatComp││Anthro ││Bedro ││Codex│
    │letions ││pic    ││ck    ││     │
    ├────────┤├───────┤├──────┤├─────┤
    │OpenAI  ││Anthro ││AWS   ││Open │
    │OpenRou ││pic 直 ││Bedro ││AI   │
    │ter     ││连     ││ck    ││Codex│
    │Gemini  ││MiniMax││      ││xAI  │
    │本地引擎││DashSc ││      ││     │
    │...     ││ope   ││      ││     │
    └────────┘└───────┘└──────┘└─────┘
```

Hermes 的解决方案是一个叫 `ProviderTransport` 的抽象基类（`agent/transports/base.py:16`）。它的核心思想很简单：不管底层 API 长什么样，对 Agent 核心来说，每次调用都只有四个步骤：

1. **convert_messages()** — 把 OpenAI 格式的消息转成 Provider 要的格式
2. **convert_tools()** — 把工具定义转成 Provider 要的格式
3. **build_kwargs()** — 组装完整的 API 调用参数
4. **normalize_response()** — 把 Provider 返回的结果转成统一的 `NormalizedResponse`

目前有四个 Transport 实现：

| Transport | 对应的 api_mode | 覆盖的 Provider |
|-----------|----------------|----------------|
| `ChatCompletionsTransport` | `chat_completions` | OpenAI、OpenRouter、Gemini、本地引擎等（默认） |
| `AnthropicTransport` | `anthropic_messages` | Anthropic 直连、第三方 Anthropic 兼容端点 |
| `BedrockTransport` | `bedrock_converse` | AWS Bedrock |
| `CodexTransport` | `codex_responses` | OpenAI Codex、xAI |

Agent 初始化时会自动检测应该用哪个 Transport（`run_agent.py:982-1013`）。检测逻辑是一串 if-elif：先看用户是否显式指定了 `api_mode`，然后根据 provider 名称或 base_url 的特征自动推断。比如 URL 以 `/anthropic` 结尾的第三方端点会自动走 `AnthropicTransport`——这是为了兼容 MiniMax、DashScope 这类使用 Anthropic 协议的国内服务。

### 这样设计的好处

新增一个 Provider 只需要实现四个方法，不需要改动 Agent 核心的任何代码。Transport 的注册是 lazy 的——直到第一次使用某个 `api_mode` 时才导入对应的模块（`agent/transports/__init__.py:14`），启动时零开销。

## 第四站：工具执行——Agent 的手脚

当模型响应包含工具调用时，Agent 需要真正去"做事"。这部分由工具层处理。

### 工具是怎么注册的

Hermes 用了一个"自注册"模式。`tools/` 目录下的每个文件（66 个）在被 import 时，会在模块级别调用 `registry.register()`：

```python
registry.register(
    name="read_file",
    toolset="file",
    schema=READ_FILE_SCHEMA,
    handler=_handle_read_file,
)
```

但 Hermes 不会在启动时盲目 import 所有工具文件。`discover_builtin_tools()`（`tools/registry.py:56`）先用 AST 静态分析扫描每个 .py 文件，只有确认包含 `registry.register()` 调用的才会 import。这避免了不必要的依赖加载——比如浏览器工具会拉 Playwright，如果用户只需要文件操作，为什么要加载浏览器？

### 串行还是并行

一次模型响应可能包含多个工具调用（比如同时搜索三个不同的关键词）。`_execute_tool_calls()`（`run_agent.py:8594`）会判断这批调用是否可以并行：

```python
if not _should_parallelize_tool_batch(tool_calls):
    return self._execute_tool_calls_sequential(...)
return self._execute_tool_calls_concurrent(...)
```

判断逻辑是：只读工具（如 `read_file`、`web_search`）总是可以并行；文件读写只有在目标路径不重叠时才能并行；其他有副作用的工具串行执行。这是一个务实的选择——完全串行太慢，完全并行可能有竞态，按副作用分类是一个合理的折中。

## 第五站：子代理——当任务太大了

有时候一个 Agent 不够用。比如"分析这三个文件然后写一份报告"——分析三个文件可以并行，但写报告要等分析完成。Hermes 的解决方案是**子代理委托**（`tools/delegate_tool.py`）。

### 安全隔离

```
┌─── 父 Agent (depth=0) ─────────────────────────┐
│                                                  │
│  工具: terminal, file, web, browser, memory,     │
│        delegate_task, clarify, send_message, ... │
│                                                  │
│  iteration_budget: 90 (共享池)                   │
│                                                  │
│       delegate_task(goal="分析文件 X")            │
│            │                                     │
└────────────┼─────────────────────────────────────┘
             │
             ▼
┌─── 子 Agent (depth=1) ──────────────────────────┐
│                                                  │
│  工具: terminal, file, web, browser              │
│  禁用: delegate_task ✗  (不能再嵌套)              │
│        clarify ✗        (没有 stdin)             │
│        memory ✗         (避免并发写)              │
│        send_message ✗   (防止副作用)              │
│        execute_code ✗   (强制逐步推理)            │
│                                                  │
│  iteration_budget: 从父池扣减                    │
│  审批: auto_deny (默认) / auto_approve (可配)    │
│                                                  │
│       delegate_task → ❌ 被拒绝 (MAX_DEPTH=1)    │
│                                                  │
└──────────────────────────────────────────────────┘
```

子代理不是父代理的完整克隆。它有严格的限制（`delegate_tool.py:41-49`）：

- **不能递归委托**（`delegate_task` 被禁用）——防止无限嵌套
- **不能和用户交互**（`clarify` 被禁用）——子代理在后台线程，没有 stdin
- **不能写持久记忆**（`memory` 被禁用）——避免并发写冲突
- **不能发消息到其他平台**（`send_message` 被禁用）——防止副作用
- **不能执行代码块**（`execute_code` 被禁用）——强制逐步推理

子代理的工具集是父代理的**子集**（取交集，`delegate_tool.py:891-930`），不可能获得父没有的能力。嵌套深度默认限制为 1 层（`delegate_tool.py:129`，`MAX_DEPTH = 1`）——也就是说，parent 可以 spawn child，但 child 不能再 spawn grandchild。

### 审批问题

子代理运行在 `ThreadPoolExecutor` 的工作线程里，没有 CLI 的 stdin 访问权。当它要执行危险操作时怎么办？默认行为是 `_subagent_auto_deny`——自动拒绝所有需要审批的操作。这很安全但保守。对于 cron 或批量运行场景，可以配置为 `_subagent_auto_approve`。

## 网关层：同一个 Agent，二十个平台

到目前为止我们一直在讲 CLI 入口。但如果消息来自 Telegram 呢？

`GatewayRunner`（`gateway/run.py:620`）是网关层的核心。它维护一个 Agent 缓存（`run.py:709`），按 `session_key`（格式：`"agent:main:<platform>:<type>:<chat_id>"`，`run.py:567`）索引。同一个聊天窗口的消息复用同一个 `AIAgent` 实例，上限 128 个（`run.py:41`，`_AGENT_CACHE_MAX_SIZE = 128`），空闲 1 小时淘汰。

每个平台适配器（如 `telegram.py`、`discord.py`）继承 `BasePlatformAdapter`（`gateway/platforms/base.py:1121`），实现三个抽象方法：`start()`、`stop()`、`send_message()`。原生平台事件（Telegram 的 `Update`、Discord 的 `Message`）被转换为统一的 `MessageEvent`，然后调用 `handle_message()`（`base.py:2221`），最终走到和 CLI 完全相同的 `AIAgent.run_conversation()`。

这就是为什么 Hermes 能"一个进程服务所有平台"——从 Agent 核心的角度看，消息从哪来是透明的。差异被封装在平台适配器里。

## 中断：当你改主意了

在整个流程的任何一步，用户都可能中断。Hermes 的中断机制是多层级的（`run_agent.py:4050-4149`）：

```
用户按 Ctrl+C / 发送新消息
         │
         ▼
  agent.interrupt(message)          ← 另一个线程调用
         │
         ├──→ _interrupt_requested = True
         │
         ├──→ 绑定到执行线程 (_execution_thread_id)
         │
         ├──→ Fan-out 到所有工具工作线程 (_tool_worker_threads)
         │         │
         │         ├──→ terminal_tool: 终止子进程
         │         ├──→ browser_tool: 中断页面操作
         │         └──→ ...
         │
         └──→ 递归传播到子代理 (_active_children)
                   │
                   └──→ child.interrupt(message)
                              └──→ ...（同样的传播链）
```

1. 另一个线程调用 `agent.interrupt(message)` → 设置 `_interrupt_requested = True`
2. 信号绑定到 Agent 的执行线程
3. 传播到所有并发工具工作线程
4. 递归传播到所有活跃子代理
5. 主循环每次迭代开头检查中断标志（`run_agent.py:9997-10003`），触发则 break

还有一个更温和的机制——`steer()`（`run_agent.py:4151`）。它不中断当前操作，而是把一段文本缓存起来，等当前工具批次完成后注入到结果中。这适合"不要打断它，但让它下一步调整方向"的场景。

## 依赖关系：严格单向

回顾整个流程，模块间的依赖是**严格单向**的：

```
CLI / 网关  →  Agent 核心  →  工具层
```

没有反向依赖。需要反向调用的地方（比如 `delegate_tool.py` 要创建新的 `AIAgent`，`gateway/run.py` 要创建 `AIAgent`）都用**函数体内 lazy import** 打破循环：

```python
# delegate_tool.py:866（在函数体内，不在模块顶部）
from run_agent import AIAgent
```

这不是偶然的——13,000 行的 `run_agent.py` 被很多模块依赖，如果允许它反向 import 工具层或网关层，就会形成循环导入，Python 会报错。Lazy import 是务实的解决方案。

## 关键设计模式总结

| 模式 | 在哪里 | 解决什么问题 |
|------|--------|------------|
| Transport ABC | `agent/transports/base.py` | 20+ 个 Provider 的 API 差异 |
| 自注册 Registry | `tools/registry.py` | 66 个工具的按需加载 |
| Lazy Import | `delegate_tool.py`、`gateway/run.py` | 打破循环依赖 |
| 系统提示缓存 | `run_agent.py:9814` | 最大化 prefix cache 命中率（Anthropic 原生支持，本地引擎也受益） |
| 记忆双注入 | 系统提示（静态）+ 用户消息（动态） | 在缓存友好和信息新鲜之间取平衡 |
| 队列化中断 | `cli.py:10824` | 主线程响应性 vs Agent 线程工作 |
| 子代理隔离 | `delegate_tool.py:41-49` | 防止并发写冲突和无限递归 |
| Agent 缓存 | `gateway/run.py:41` | 128 个 session 复用 Agent 实例 |

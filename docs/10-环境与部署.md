# 10 - 环境与部署：从笔记本到云集群

## 一个 Agent 的栖息地

前面九篇聚焦 Hermes 的内部系统——Agent 核心、工具、技能、插件、网关。但一个 Agent 要运行，总得有一个地方"住"。是在你的 Mac 上本地跑？Docker 容器里隔离运行？远程服务器上通过 SSH 执行命令？还是在 Modal 云端按秒计费？

Hermes 的回答是：**都可以**——而且你可以在不改代码的前提下切换。

## 六种终端后端

Agent 执行命令（`terminal` 工具）的底层环境由终端后端决定。`cli-config.yaml.example:148-237` 定义了六种：

| 后端 | 适用场景 | 隔离级别 | 特点 |
|------|---------|---------|------|
| `local` | 日常开发 | 无（直接在宿主机执行） | 默认，最快，零配置 |
| `docker` | 可复现环境 | 容器级 | 支持挂载工作目录，容器内审批直接放行 |
| `ssh` | 远程服务器 | 网络级 | 配置 host/user/port/key |
| `singularity` | HPC 集群 | 容器级（无 root） | 适合学术计算环境 |
| `modal` | GPU 云/Serverless | 云级 | 按需启动，空闲自动休眠 |
| `daytona` | 云开发环境 | 云级 | 类似 GitHub Codespaces，支持磁盘持久化 |

后四种都是容器化或远程环境，Agent 在这些环境里执行命令时**自动跳过安全审批**（`approval.py:889-891`）——因为容器/沙箱本身就是安全边界，不需要额外确认。

通用资源限制可配（适用于 docker/singularity/modal/daytona）：CPU 核数、内存（默认 5GB）、磁盘（默认 50GB）、是否持久化。

## Docker 部署

`Dockerfile`（67 行）是一个三阶段构建：

```
Stage 1 (uv_source):   提取 uv/uvx 二进制
Stage 2 (gosu_source):  提取 gosu（用于运行时 UID 切换）
Stage 3 (debian:13.4):  最终运行时镜像
```

几个设计选择值得注意：

**层缓存优化**。先复制 `package.json` / `package-lock.json` 安装 npm 依赖，再复制源码——源码变化时 npm 层不重建。Python 依赖通过 `uv pip install`（Rust 写的包管理器）安装，比 pip 快一个数量级。

**非 root 运行**。创建 `hermes` 用户（UID 10000），运行时通过 `HERMES_UID` 环境变量可覆盖——`docker/entrypoint.sh` 用 `gosu` 切换到指定 UID。这让容器内文件权限和宿主机对齐，避免 volume 挂载的权限问题。

**PID 1 管理**。Entrypoint 用 `tini`（一个微型的 init 进程）包装，确保孤儿进程（以 MCP stdio 子进程和 git 为典型）被正确收割，不会堆积成僵尸进程。

`docker-compose.yml` 定义两个服务：`gateway`（消息网关）和 `dashboard`（Web 管理界面），均用 `network_mode: host` 和 `~/.hermes:/opt/data` volume 挂载。

## 安装向导

`setup-hermes.sh` 是一键安装脚本（`curl -fsSL ... | bash`），流程是：

1. 检测平台（Linux/macOS/Android Termux）
2. 安装 `uv` 包管理器
3. 确保 Python 3.11+ 可用
4. 创建虚拟环境
5. 安装依赖（优先用 `uv sync --locked` 带哈希验证，失败降级到 `uv pip install`）
6. 可选安装 `ripgrep`（全文搜索工具）
7. 创建 `hermes` 命令的 symlink
8. 同步预置技能
9. 可选运行交互式配置向导

Termux（Android）有单独的依赖路径——`.[termux]` extra 排除了不兼容 Android 的语音依赖。

## 多 Profile 机制

一台机器上可能需要运行多个 Hermes 实例——比如工作和个人用不同的 API key 和记忆。`hermes --profile <name>` 切换 profile，每个 profile 是一个独立的 `~/.hermes/profiles/<name>/` 目录，包含自己的配置、记忆、会话、技能和 cron 任务（`hermes_cli/profiles.py`）。

`default` profile 就是 `~/.hermes` 本身，零迁移。创建新 profile 时可以选择克隆现有配置（`clone_config=True`）或完全独立。`hermes profile export/import` 通过 tar.gz 导入导出（有路径穿越防护）。

一个便利设计：创建 profile 时自动在 `~/.local/bin/` 生成 wrapper 脚本——如果你创建了名为 `work` 的 profile，就可以直接运行 `work` 命令（等价于 `hermes --profile work`）。

## Nix 和 Homebrew

除了 pip/uv 安装，Hermes 还提供了 Nix flake（`flake.nix`）和 Homebrew formula（`packaging/homebrew/hermes-agent.rb`）。

Nix flake 通过 `uv2nix` 将 Python 依赖转化为 Nix 派生，导出三个包（default/tui/web），支持 NixOS module 声明式部署。Homebrew formula 用 `Language::Python::Virtualenv` mixin，为三个入口点（hermes/hermes-agent/hermes-acp）创建 env_script wrapper。

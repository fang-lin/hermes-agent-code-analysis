# 03 - 工具系统：Agent 的 66 双手

## Agent 只会思考，工具让它能做事

在前两篇中，我们看到 AIAgent 的核心循环是"调用模型 → 解析响应 → 执行工具 → 循环"。模型负责决策（"我应该搜索一下网页"），但真正去搜索的不是模型——而是工具系统。没有工具，Agent 就只是一个会聊天的 API 客户端。

Hermes 的工具系统由三层组成：

```
┌───────────────────────────────────────────────────────┐
│  model_tools.py — 调度层                               │
│  接收模型的工具调用请求 → 查表 → 分发给对应 handler     │
├───────────────────────────────────────────────────────┤
│  tools/registry.py — 注册层                            │
│  ToolRegistry 单例，持有所有工具的 schema + handler     │
│  AST 扫描发现 → 按需 import → 模块级自注册             │
├───────────────────────────────────────────────────────┤
│  tools/*.py (66 个文件) — 实现层                       │
│  每个文件定义 schema + handler + register() 调用       │
│  + toolsets.py 管理分组                                │
│  + approval.py / path_security / url_safety 安全拦截   │
└───────────────────────────────────────────────────────┘
```

## 一个工具长什么样

理解整个工具系统之前，先看一个具体的工具定义。以 `read_file` 为例（`tools/file_tools.py`）：

**第一步，定义 Schema**（`file_tools.py:1025-1037`）——这是告诉模型"这个工具叫什么、接收什么参数"的描述，使用 OpenAI function-calling 的标准 JSON 格式：

```json
{
  "name": "read_file",
  "description": "Read a text file with line numbers...",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {"type": "string"},
      "offset": {"type": "integer", "default": 1},
      "limit": {"type": "integer", "default": 500, "maximum": 2000}
    },
    "required": ["path"]
  }
}
```

**第二步，实现 Handler**（`file_tools.py:1089-1091`）——一个普通的 Python 函数，接收 `args` 字典和关键字参数，返回 JSON 字符串：

```python
def _handle_read_file(args, **kw):
    tid = kw.get("task_id") or "default"
    return read_file_tool(path=args.get("path", ""), ...)
```

所有工具的返回值必须是 JSON 字符串。`registry.py` 提供了两个辅助函数（`registry.py:456-482`）：`tool_result(data)` 返回成功结果，`tool_error(message)` 返回错误。

**第三步，注册**（`file_tools.py:1118-1121`）——在模块级别调用 `registry.register()`：

```python
registry.register(
    name="read_file", toolset="file",
    schema=READ_FILE_SCHEMA, handler=_handle_read_file,
    check_fn=_check_file_reqs, emoji="📖",
    max_result_size_chars=float('inf')
)
```

这三步就是 Hermes 所有 66 个工具遵循的模式。为什么选择 OpenAI 的 function-calling 格式而非自定义格式？因为这是行业事实标准——所有主流 LLM（OpenAI、Anthropic、Gemini）都理解这种 schema。Hermes 的 Transport 层会在必要时将其转换为各 Provider 的原生格式（比如 Anthropic 的 `input_schema` 格式），但**工具定义本身只需要写一次**。

## 工具是怎么被发现和加载的

66 个工具文件不是在启动时全部 import 的——那会拉入 Playwright、faster-whisper 等重依赖，拖慢启动速度。`discover_builtin_tools()`（`tools/registry.py:56`）用了一个巧妙的两阶段加载：

```
阶段 1：AST 扫描（不执行代码）
  tools/*.py → 解析语法树 → 有 registry.register() 调用？
  ├─ 是 → 标记为"需要 import"
  └─ 否 → 跳过（如 registry.py 自身、__init__.py）

阶段 2：按需 import（只对标记的文件）
  import tools.file_tools  → 触发模块级 register() → 工具注册完毕
  import tools.web_tools   → ...
  import tools.browser_tool → 拉入 Playwright（只有用到才付这个代价）
```

为什么用 AST 扫描而非简单的"import 所有文件然后 try-except"？因为 import 会执行模块级代码，可能有副作用（比如建立数据库连接、启动后台进程）。AST 只解析语法树不执行代码，是零副作用的发现机制。如果 AST 扫描本身失败（比如文件有语法错误），该文件被静默跳过并记录 warning（`registry.py:70`），不会影响其他工具的加载。

这个机制由 `model_tools.py:139` 在模块级触发，时机是 Agent 初始化时。

工具加载完毕，下一个问题就是：这些工具应该全部暴露给模型吗？

## 工具集：同一批工具，不同场景

并非所有场景都需要所有工具。CLI 用户可能需要浏览器工具，但 Telegram 群聊不需要（甚至不应该有——想象一个 Agent 在群里自动打开浏览器访问网页）。

`toolsets.py` 解决这个问题。它定义了一个**共享核心**加**平台扩展**的模型：

```
_HERMES_CORE_TOOLS（38 个工具）── 所有平台共享
    │
    ├─ hermes-cli        = 核心工具
    ├─ hermes-telegram   = 核心工具
    ├─ hermes-discord    = 核心工具 + Discord 管理工具
    ├─ hermes-feishu     = 核心工具 + 飞书特有工具
    ├─ hermes-yuanbao    = 核心工具 + 元宝特有工具
    ├─ hermes-acp        = 核心工具 - clarify/send_message 等
    └─ ...
```

核心工具列表（`toolsets.py:31-63`）包含 38 个工具：web_search、terminal、read_file、write_file、patch、search_files、browser 系列、memory、session_search、delegate_task、skills 系列等。这份列表是**单一来源**——修改一处，所有平台同步更新。

每个 toolset 支持三个字段：`tools`（直接列表）、`includes`（引用其他 toolset）、`description`。支持递归组合——`resolve_toolset()`（`toolsets.py:529-579`）做 DFS 展开并检测环路，防止 A includes B includes A 的死循环。

用户可以通过 `config.yaml` 的 `platform_toolsets` 覆盖任何平台的工具集，也可以通过 `--toolsets` CLI 参数临时指定。如果某个工具的依赖不满足（比如没装 Playwright），工具的 `check_fn` 在注册时就会返回 False，该工具被静默从可用列表中移除。

还有一个特殊用途的 `toolset_distributions.py`（`toolset_distributions.py:1-19`）——它不是给日常使用的，而是为 `batch_runner.py` 的数据生成场景设计的。它定义了 17 种概率分布（如 `research` 分布中 web 工具 90% 激活、browser 70%），用于在批量轨迹生成时随机变化工具组合，模拟真实世界中用户使用不同工具集的多样性。

## 安全：三道防线

工具系统最敏感的部分是安全——Agent 执行的命令可能删除文件、修改系统配置、发起网络请求。Hermes 在工具调用路径上设了三道防线。

### 第一道：命令审批（approval.py）

这是最复杂的安全机制，位于工具层和终端执行之间。`check_all_command_guards()`（`approval.py:880`）是入口，它按优先级执行以下检查：

```
命令进入
  │
  ├─ 容器化环境？(docker/singularity/modal/daytona)
  │     └─ 是 → 直接放行（容器本身就是沙箱）
  │
  ├─ 硬核封锁？(HARDLINE_PATTERNS, approval.py:143-165)
  │     └─ 匹配 → 无条件拒绝（rm -rf /、mkfs、fork bomb 等 12 类）
  │
  ├─ --yolo 或 approvals.mode=off？
  │     └─ 是 → 放行
  │
  ├─ Tirith 内容扫描（外部 Rust 二进制）
  │     └─ 检测同质字形 URL、管道注入等内容级威胁
  │
  ├─ 危险模式匹配？(DANGEROUS_PATTERNS, approval.py:201-267)
  │     └─ 匹配 47 个正则（rm -rf、chmod 777、DROP TABLE、
  │        git reset --hard、curl|sh 等）
  │
  └─ 需要审批
       ├─ smart 模式 → 辅助 LLM 自动判断 APPROVE/DENY/ESCALATE
       ├─ CLI → 交互式提示 [o]nce/[s]ession/[a]lways/[d]eny
       └─ Gateway → 异步等待用户响应（最长 300 秒）
```

硬核封锁（`HARDLINE_PATTERNS`）和 `--yolo` 模式不能共存——即使开了 `--yolo`，`rm -rf /` 仍然被拒绝。这是 Hermes 的安全底线。

审批结果有四种持久化级别：`once`（仅本次）、`session`（本次会话内）、`always`（写入配置，后续同类命令不再询问）、`deny`（拒绝执行）。`always` 级别的审批结果保存在 `config.yaml` 的 `command_allowlist` 中（`approval.py:523-529`）。

`smart` 模式（`approval.py:703-747`）是一个有趣的设计——用一个辅助 LLM 来判断命令是否安全，返回 APPROVE、DENY 或 ESCALATE（需要人工介入）。这在 Gateway 场景下特别有用——如果每个可能危险的命令都要等用户在 Telegram 上回复确认，Agent 的效率会很低。smart 模式相当于一个 AI 安保员，能自动放行大部分安全操作，只对真正可疑的操作才打扰用户。

### 第二道：路径和 URL 安全

**路径安全**（`tools/path_security.py:15-34`）防止路径遍历攻击（通过构造 `../../etc/passwd` 这样的路径来访问允许目录之外的文件）。`validate_within_dir()` 用 Python 的 `path.resolve().relative_to(root)` 确保工具操作的文件不会逃逸出允许的目录。被技能管理、cron 任务、凭证文件操作等模块使用。

**URL 安全**（`tools/url_safety.py:1-231`）防止 SSRF（Server-Side Request Forgery）。核心是 `is_safe_url()`——对 URL 做 DNS 解析后检查 IP，**fail-closed**（DNS 失败则拒绝）。特别值得注意的是，它永久封锁了所有云平台的元数据端点：

- `169.254.169.254`（AWS/GCP 元数据服务）
- `metadata.google.internal`
- Azure、DigitalOcean 等元数据 IP

即使配置了 `security.allow_private_urls: true` 放开私网 IP，元数据端点仍然不可访问（`url_safety.py:162`）。这意味着即使 Agent 在云服务器上运行，它也不能通过工具调用读取 VM 的 IAM 凭证——这是一个实际的攻击向量。

### 第三道：Tirith 内容级扫描

Tirith（`tools/tirith_security.py:1-692`）是一个外部的 Rust 写的安全扫描器，检测正则表达式无法捕获的内容级威胁（比如使用 Unicode 同质字形伪装的恶意 URL）。它作为独立二进制运行，首次使用时从 GitHub releases 自动下载，并验证 SHA-256 校验和和 cosign 供应链签名（一种验证软件包来源未被篡改的标准机制）（`tirith_security.py:282-383`）——下载在后台线程进行，不阻塞 Agent 启动。

默认 `fail_open=true`——如果 Tirith 不可用（未安装、下载失败），命令照常执行。这是一个务实的选择：大多数用户不需要内容级扫描，强制要求会增加部署复杂度。对于高安全场景，可以配置 `security.tirith_fail_open: false` 切换为 fail-closed。

## MCP：让外部工具成为一等公民

Hermes 的工具不局限于内置的 66 个——通过 MCP（Model Context Protocol），任何实现了 MCP 协议的外部服务都能注册为 Hermes 的工具。

MCP 集成的实现（`tools/mcp_tool.py`）运行在一个专用的后台事件循环（`mcp_tool.py:55-69`）上，支持两种传输方式：`stdio`（子进程通信）和 `streamablehttp`（HTTP 长连接）。每个 MCP server 的生命周期由 asyncio Task 管理。

关键的设计决策是：**MCP 工具和内置工具使用完全相同的注册接口**。`discover_mcp_tools()`（`mcp_tool.py:2792-2838`）在启动时连接配置的 MCP servers，获取它们提供的工具列表，然后用同一个 `registry.register()` 注册进 ToolRegistry。从 Agent 的角度看，MCP 工具和内置工具没有任何区别——都是 schema + handler 的组合。

区别在 toolset 命名上：MCP 工具的 toolset 以 `mcp-` 开头（如 `mcp-filesystem`），便于用户按需启用禁用。注册时有重名保护（`registry.py:194-213`）：MCP 工具之间可以互相覆盖（后注册的赢），但 MCP 工具不能覆盖内置工具（反之亦然）。

当 MCP server 的工具列表发生变化（比如服务端升级新增了工具），server 会发送 `tools/list_changed` 通知，Hermes 自动做 `deregister() + 重新注册`（`registry.py:229-252`），无需重启 Agent。

安全方面：MCP 子进程只继承一组白名单环境变量（`_SAFE_ENV_KEYS`：PATH、HOME、USER 等，`mcp_tool.py:252-267`），敏感凭证不会泄露到外部进程。

安全解决了"谁能调用什么"的问题。接下来是另一个治理维度——调用之后，结果能有多大？

## 工具结果的大小治理

一个实际问题：如果工具返回了 1MB 的文件内容，全部塞进消息历史会怎样？上下文窗口爆炸、API 费用飙升、后续请求延迟变高。

Hermes 用三层机制控制工具结果大小（`tools/tool_result_storage.py:1-23`）：

```
层 1: 工具内部预截断
  │  终端输出 ≤ 50K chars (tool_output_limits.py:39)
  │  文件读取 ≤ 2000 行、每行 ≤ 2000 chars
  │  全部可通过 config.yaml 调整
  ▼
层 2: per-result 持久化 (tool_result_storage.py:116-172)
  │  单个结果超过阈值（默认 100K chars）？
  │  ├─ 是 → 完整内容写入 /tmp/hermes-results/{id}.txt
  │  │       消息中替换为 <persisted-output> 标记 + 1500 chars 预览
  │  │       模型可用 read_file 按需读取全文
  │  └─ 否 → 保持原样
  ▼
层 3: per-turn 总预算 (tool_result_storage.py:175-226)
  │  一个轮次内所有工具结果加起来超过 200K chars？
  │  └─ 是 → 按大小降序，将最大的结果持久化，直到总量降到预算内
  ▼
最终的 messages 列表
```

这个设计的巧妙之处在于**不丢弃信息**。传统做法是简单截断超大结果（丢掉后面的内容），但 Hermes 选择"持久化 + 按需读取"——完整内容保存在沙箱文件系统里，消息中只放预览和路径，模型如果需要完整内容可以用 `read_file` 工具读取。代价是多一次工具调用，但好处是不会无声地丢失信息。

有一个特殊处理：`read_file` 工具的 `max_result_size_chars` 被钉死为 `float('inf')`（`budget_config.py:12-14`）——因为如果 read_file 的结果也被持久化，模型就需要用 read_file 去读 read_file 的持久化结果，陷入无限循环。

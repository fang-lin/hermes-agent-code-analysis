# 11 - 批量运行与 RL：从 Agent 到训练数据

## Agent 不只是产品，也是数据工厂

Nous Research 做 Hermes Agent 不只是为了卖一个 AI 助手——他们还在用它生成训练下一代工具调用模型的数据。这就是 Hermes 的"研究就绪"（research-ready）定位：Agent 在日常使用中产生的对话轨迹，经过压缩和格式化后，可以直接用于监督微调（SFT）或强化学习（GRPO/PPO）训练。

这不是后加的功能——从 `save_trajectories` 参数（[02-Agent 核心](02-agent核心.md)）到 `toolset_distributions`（[03-工具系统](03-工具系统.md)），研究需求从一开始就渗透在 Hermes 的设计中。本章聚焦三个核心组件：批量轨迹生成器、轨迹压缩器、RL 训练环境。

## BatchRunner：大规模轨迹工厂

`batch_runner.py`（`BatchRunner` 类，第 515 行）从 JSONL 数据集批量生成对话轨迹。每行数据包含一个 `prompt`（任务描述），可选的 `image`（容器镜像覆盖）和 `cwd`（工作目录）。

```
输入: prompts.jsonl (每行一个任务)
  │
  ├─ multiprocessing.Pool (N workers)
  │    ├─ Worker 1: AIAgent + 工具集 → trajectory
  │    ├─ Worker 2: AIAgent + 工具集 → trajectory
  │    └─ Worker N: ...
  │
  ├─ 每 batch 输出 batch_{n}.jsonl
  └─ 最终合并为 trajectories.jsonl
```

几个设计亮点：

**工具集随机化**。通过 `toolset_distributions.py`（[03-工具系统](03-工具系统.md) 介绍过）的 17 种概率分布，每次运行随机激活不同工具集——模拟真实用户使用不同工具组合的多样性，让训练数据覆盖更广。

**断点续传**（`batch_runner.py:796-839`）。`--resume` 时扫描已完成批次文件，通过内容匹配（而非索引）跳过已处理的 prompt——这比基于行号的续传更健壮，能容忍数据集中间插入新行。

**推理过滤**。零推理轨迹（模型没有使用 `<REASONING_SCRATCHPAD>` 或原生推理能力的对话）被自动丢弃（`batch_runner.py:444-448`）——对于训练工具调用模型来说，没有推理过程的轨迹价值有限。

**工具统计**。每条轨迹自动提取工具调用的 count/success/failure 统计，并对所有可能的工具名补全零值（`batch_runner.py:114-194`），确保 Parquet schema 一致——这是为下游训练管线的 DataFrame 处理准备的。

## 轨迹压缩器：让数据更紧凑

模型的上下文窗口和训练预算都有限——一条 50K token 的轨迹放进 15K token 的训练窗口需要压缩。`trajectory_compressor.py` 做的就是这件事。

压缩策略（`trajectory_compressor.py:709`，`compress_trajectory()`）遵循一个清晰的原则——**保头保尾压中间**：

1. 保护头部：第一条 system/human/gpt/tool 消息不动
2. 保护尾部：最后 N 轮（默认 4）不动
3. 中间部分：用摘要模型（默认 `gemini-3-flash-preview`）生成压缩摘要
4. 用单条 `[CONTEXT SUMMARY]:` 消息替换被压缩区间

为什么保护头尾？头部包含任务定义和初始上下文——模型需要知道"要做什么"。尾部包含最终结果和成功/失败状态——这是训练信号的核心。中间是"怎么做到的"过程，摘要化后信息损失可接受。

压缩是异步并行的（`trajectory_compressor.py:829`），用 `asyncio.Semaphore(50)` 控制并发 API 调用，300 秒超时保护。配置通过 `CompressionConfig` 类（第 83 行）指定目标 token 数（默认 15,250）和摘要 token 预算（默认 750）。

## RL 训练环境

`environments/` 目录实现了 Atropos RL 框架的环境适配，让 Hermes 的 Agent 能力可以在强化学习中被训练和评估。

```
atroposlib.BaseEnv (外部框架)
    └── HermesAgentBaseEnv (hermes_base_env.py)
           ├── TerminalTestEnv     — 基础终端任务
           ├── HermesSweEnv        — 软件工程任务
           └── benchmarks/
                ├── TerminalBench2  — 89 个终端任务（Modal 沙箱）
                ├── Tblite          — 100 个校准任务
                └── YCBench         — 长视野战略基准
```

训练分两个阶段：

- **Phase 1**（评估/SFT）：服务器端解析工具调用，适合生成监督训练数据
- **Phase 2**（GRPO/PPO）：客户端解析工具调用，获取真实 token IDs 和 logprobs——这是梯度更新需要的

工具调用格式因模型而异——Hermes 格式、Mistral 格式、Llama3 JSON、Qwen、DeepSeek 等各有自己的约定。`tool_call_parsers/` 目录重新实现了 11 种格式的解析器，确保无论训练什么模型都能正确提取工具调用。

`rl_cli.py` 是 RL 训练的命令行入口，与 `tinker-atropos/` 子模块集成。它的 Agent 配置比日常使用有更高的迭代上限——`RL_MAX_ITERATIONS = 200`（日常默认 90），系统提示包含完整的 RL 工作流指引（发现→检查→创建→配置→测试→训练→评估）。

## 数据生成配置

`datagen-config-examples/` 目录提供了几种典型配置模板：

- `trajectory_compression.yaml` — 轨迹压缩配置，目标 29K token
- `web_research.yaml` — Web 研究任务批量生成，4 workers
- `example_browser_tasks.jsonl` — 浏览器任务示例数据集

这些配置展示了 Hermes 作为训练数据工厂的使用方式——不是手动标注，而是让 Agent 自己执行任务，自动记录轨迹，再经压缩后喂给训练管线。
